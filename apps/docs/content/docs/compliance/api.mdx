The `complianced` server exposes a REST API on port `8091`. All request and response bodies are JSON. The server uses only the Go standard library for HTTP routing.

## Authentication

All endpoints require the `X-Api-Key` header, except:

- `GET /healthz` -- health check
- `POST /v1/kyc/webhook/{provider}` -- provider webhook callbacks

If `COMPLIANCE_API_KEY` is not set, authentication is disabled.

## Applications

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/v1/applications` | Create a new application |
| `GET` | `/v1/applications` | List applications (filter with `?status=`) |
| `GET` | `/v1/applications/stats` | Application statistics by status |
| `GET` | `/v1/applications/{id}` | Get application by ID |
| `PATCH` | `/v1/applications/{id}` | Update application (draft save) |
| `POST` | `/v1/applications/{id}/submit` | Submit application for review |

### Create Application

```bash
curl -X POST http://localhost:8091/v1/applications \
  -H "X-Api-Key: $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"given_name":"Alice","family_name":"Smith","email":"alice@example.com"}'
```

Response (201):

```json
{
  "id": "a1b2c3d4...",
  "status": "draft",
  "given_name": "Alice",
  "family_name": "Smith",
  "created_at": "2026-03-17T00:00:00Z"
}
```

### Submit Application

Only applications in `draft` status can be submitted. Returns `409 Conflict` otherwise.

```bash
curl -X POST http://localhost:8091/v1/applications/{id}/submit \
  -H "X-Api-Key: $API_KEY"
```

## KYC Verification

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/v1/kyc/verify` | Initiate KYC verification |
| `GET` | `/v1/kyc/status/{verificationId}` | Check verification status |
| `GET` | `/v1/kyc/application/{applicationId}` | List verifications for an application |
| `POST` | `/v1/kyc/webhook/{provider}` | Receive provider webhook (no auth) |

### Initiate Verification

```bash
curl -X POST http://localhost:8091/v1/kyc/verify \
  -H "X-Api-Key: $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "application_id": "a1b2c3d4...",
    "given_name": "Alice",
    "family_name": "Smith",
    "email": "alice@example.com",
    "provider": "jumio"
  }'
```

Response (201):

```json
{
  "verification_id": "v-abc123...",
  "provider": "jumio",
  "status": "pending",
  "redirect_url": "https://netverify.com/...",
  "created_at": "2026-03-17T00:00:00Z"
}
```

### Verification Status Values

| Status | Description |
|--------|-------------|
| `pending` | Verification in progress |
| `approved` | Identity verified |
| `declined` | Verification failed |
| `expired` | Session expired |
| `error` | Provider error |

### Webhook

Providers send results to `/v1/kyc/webhook/{provider}`. The handler validates the HMAC-SHA256 signature, deduplicates events, and updates KYC status. Signature headers per provider:

| Provider | Signature Header |
|----------|-----------------|
| Jumio | `X-Jumio-Signature` |
| Onfido | `X-SHA2-Signature` |
| Plaid | `Plaid-Verification` |

## AML Screening

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/v1/aml/screen` | Screen individual against sanctions/PEP lists |
| `POST` | `/v1/aml/monitor` | Monitor a transaction against rules |
| `GET` | `/v1/aml/alerts` | List alerts (filter with `?status=`) |

### Screen Individual

```bash
curl -X POST http://localhost:8091/v1/aml/screen \
  -H "X-Api-Key: $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"name":"John Doe","country":"US","date_of_birth":"1980-01-15"}'
```

Response:

```json
{
  "id": "scr-abc123...",
  "risk_level": "low",
  "matches": [],
  "lists_checked": ["ofac_sdn","eu_sanctions","uk_hmt","pep","adverse_media"],
  "screened_at": "2026-03-17T00:00:00Z"
}
```

### Monitor Transaction

```bash
curl -X POST http://localhost:8091/v1/aml/monitor \
  -H "X-Api-Key: $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"account_id":"acct-1","amount":15000,"currency":"USD","country":"US"}'
```

If rules trigger, an alert is generated with status `open`.

## Payments

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/v1/payments/validate` | Validate payin or payout compliance |

### Validate Payment

```bash
curl -X POST http://localhost:8091/v1/payments/validate \
  -H "X-Api-Key: $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "pay-1",
    "direction": "payout",
    "amount": 12000,
    "currency": "USD",
    "country": "US",
    "account_id": "acct-1",
    "type": "wire",
    "originator_name": "Alice Smith",
    "beneficiary_name": "Bob Jones"
  }'
```

Response includes `decision` (`approve`, `decline`, or `review`), `requires_ctr`, `requires_sar`, and `travel_rule` compliance status.

## Regulatory

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/v1/regulatory/{jurisdiction}` | Get jurisdiction requirements and limits |

Supported jurisdiction codes: `US`, `UK`, `IM` (Isle of Man).

```bash
curl http://localhost:8091/v1/regulatory/US -H "X-Api-Key: $API_KEY"
```
