## Overview

The Lux Network implements a **multi-chain, multi-consensus architecture** where each chain type runs a purpose-built Virtual Machine (VM) optimized for its workload. The node (`luxd`) manages all chains, routing messages between them and driving consensus.

There are **14 built-in chain types** across 3 tiers. The primary network chains (P, X, C) are always running. Extended chains (Q, A, B, T, Z, G, D, K, O, R, I) require the `allvms` build tag. Additionally, several infrastructure VMs (ProposerVM, RPCChainVM, DA, FalconFX, ServiceNodeVM, ChainAdapter) provide cross-cutting services.

All VMs implement the `chain.ChainVM` interface from `github.com/luxfi/vm/chain` (for linear chains) or the DAG vertex interface (for DAG chains like X-Chain). Each VM is registered via a `Factory` that satisfies `github.com/luxfi/node/vms.Factory`.

### Tech Stack

- **Language**: Go 1.26.1
- **Module**: `github.com/luxfi/node`
- **Consensus**: Quasar hybrid finality (BLS + Ringtail post-quantum threshold signatures)
- **VM Interface**: `github.com/luxfi/vm` v1.0.39
- **Consensus Package**: `github.com/luxfi/consensus` v1.22.69
- **Crypto**: `github.com/luxfi/crypto` v1.17.42 (BLS, ML-DSA, ML-KEM, secp256k1, Ringtail)

### When to Use

- Understanding which chain handles which workload in the Lux ecosystem
- Selecting the right chain for a new application or integration
- Building or extending a VM for a custom chain type
- Debugging cross-chain interactions
- Configuring `luxd` build tags for extended chain support

### Hard Requirements

1. **ALWAYS** use `github.com/luxfi/*` packages -- **NEVER** `go-ethereum` or `ava-labs`
2. Consensus is **Quasar** -- NEVER refer to it as Snow, Snowball, or Avalanche consensus
3. **NEVER** bump Go packages above v1.x.x (except `luxfi/lattice/v7` and `luxfi/zapdb/v4`)
4. Build with `-tags=allvms` to enable extended chains beyond P/X/C

## Build Tags

| Build | Chains Included | Binary Size |
|-------|-----------------|-------------|
| Default (no tag) | P, X, C (plugin), ProposerVM | ~30 MB |
| `-tags=allvms` | All 14 chain types + infrastructure VMs | ~36 MB |
| `-p core` profile | P, X, C only, plugin-ready | ~25 MB |
| `-p full` profile | All chains + gRPC + ZAP, all features | ~36 MB |

```bash
# Minimal build (3 core chains: P, X, C)
go build ./...

# Full build (3 core + 11 optional = 14 chains)
go build -tags=allvms ./...
```

The `optionalVMCount` constant is 0 in minimal builds and 11 in `allvms` builds. See `node/vms_allvms.go` and `node/vms_minimal.go`.

### Q-Chain (Quantum Chain)

| Property | Value |
|----------|-------|
| VM | QuantumVM (package `qvm`) |
| VM ID | `quantumvm` |
| Package | `github.com/luxfi/node/vms/quantumvm` |
| Consensus | Linear + Quasar PQ-BFT (hybrid BLS + Ringtail) |
| Version | 1.0.0 |

**Purpose**: The Q-Chain is the post-quantum cryptography chain. It runs the Quasar consensus engine as a first-class component, producing quantum-safe finality certificates using dual BLS + Ringtail threshold signatures. It serves as the quantum-safe anchor for the entire Lux network.

**Key Features**:
- **Quasar hybrid finality engine** (BLS + Ringtail in parallel)
- Quantum stamps (post-quantum finality proofs)
- Ringtail key generation and management (Ring-LWE based)
- ML-DSA quantum signatures (NIST FIPS 204)
- Post-quantum transaction signing
- Merkle root anchoring from other chains
- Parallel batch verification (configurable batch size, max 100 parallel verifications)

**Quasar Engine** (`quasar.go`):

The Quasar engine wraps `github.com/luxfi/consensus/protocol/quasar` and produces **dual signatures per block**:

```
BlockSigs {
    BLS      *quasar.BLSSignature      // Classical, fast (96 bytes)
    Ringtail *quasar.RingtailSignature  // Post-quantum, Ring-LWE threshold
}
```

Blocks are NOT considered produced without BOTH thresholds being met:
- **BLS Path**: Aggregate BLS signatures from 2/3+ validator weight (fast)
- **Ringtail Path**: Post-quantum threshold signatures (t-of-n, 2-round protocol)
- Both paths execute **in parallel** -- either can complete first

**Signature Types**:
```go
SignatureTypeBLS      = 0  // Classical BLS aggregate
SignatureTypeRingtail = 1  // Post-quantum threshold
SignatureTypeQuasar   = 2  // Hybrid BLS + Ringtail
SignatureTypeMLDSA    = 3  // ML-DSA fallback
```

**Subpackages**:
- `quantum/` -- Quantum-safe cryptographic primitives
- `config/` -- Q-Chain configuration

**Configuration**:
- `QuantumAlgorithmVersion` -- Algorithm version
- `RingtailKeySize` -- Ringtail key size
- `QuantumStampWindow` -- Stamp validity window
- `QuantumSigCacheSize` -- Signature cache size

## Consensus Architecture

### Consensus Engines

All chains use the Quasar consensus framework, but through different engine types:

| Engine | Type | Used By | Description |
|--------|------|---------|-------------|
| **Linear** | Block-based chain | P, C, Q, A, B, T, Z, G, D, K, O, R, I | Blocks form a single chain. Wrapped by ProposerVM (LP-181). |
| **DAG** | Vertex-based graph | X | Vertices can have multiple parents. Higher throughput via parallel processing. |
| **Quasar** | Hybrid BLS + Ringtail | Q (first-class), all (via luxd) | Dual threshold signatures for quantum-safe finality. |
| **ZAP** | Agent consensus bridge | (bridge layer) | W3C DID-based validator identity, agentic consensus. |

### Quasar Finality Flow

```
Validator proposes block
  -> Other validators verify block
  -> Cast votes (Chits/Vote messages)
  -> Consensus engine collects votes, runs polls
  -> BLS threshold met (2/3+ validator weight)     [parallel]
  -> Ringtail threshold met (t-of-n post-quantum)  [parallel]
  -> Block accepted (quantum finality achieved)
```

## Cross-Chain Communication

Chains communicate through several mechanisms:

| Mechanism | Chains | Direction | Description |
|-----------|--------|-----------|-------------|
| **Atomic Transactions** | P <-> X <-> C | Bidirectional | Shared memory transfers (primary network only) |
| **Warp Messaging** (LP-118) | Any <-> Any | Bidirectional | BLS-attested cross-chain messages |
| **Relay Chain** | Any -> R -> Any | Via R-Chain | Reliable message delivery with sequence tracking |
| **Bridge Chain** | External <-> B <-> Internal | Via B-Chain | MPC-signed cross-chain bridge with fee/limits |
| **Threshold Signing** | Any -> T | Request/Response | On-demand MPC signing service (quota-limited) |

## Troubleshooting

| Problem | Cause | Fix |
|---------|-------|-----|
| Extended chains not available | Missing build tag | Build with `-tags=allvms` |
| Q-Chain blocks not finalizing | Ringtail threshold not met | Need t+1 validators with Ringtail keys |
| D-Chain slow blocks | Block interval too high | Adjust `blockInterval` in dexvm config |
| A-Chain attestation failures | TEE not configured | Enable SGX/SEV-SNP/TDX or nvtrust on provider |
| B-Chain bridge stuck | MPC threshold not met | Need t+1 MPC parties online |
| T-Chain `ErrQuotaExceeded` | Per-chain quota exhausted | Increase quota or wait for daily reset |
| Z-Chain proof verification timeout | Complex circuit | Increase `ProofVerificationTimeout` |
| K-Chain parallel ops blocked | Max operations reached | Wait for current ops to complete (max 100) |
| G-Chain query timeout | Deep nested query | Reduce `MaxQueryDepth` or simplify query |
| I-Chain DID not found | Identity not registered | Create identity first via I-Chain |
| O-Chain stale observation | Old data submitted | Check `ObservationWindow` and resubmit |
| R-Chain relay timeout | Message delivery too slow | Increase relay timeout or check destination chain |
| Cross-chain message not delivered | Insufficient Warp sigs | Need 67%+ validator weight to attest |
| `KeyVM uses NewDefaultFactory()` error | Wrong factory init | Use `keyvm.NewDefaultFactory()` not `&Factory{}` |
