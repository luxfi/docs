Setup a Devnet Using Lux-CLI

This page demonstrates how to setup a Devnet of cloud-based validators using Lux-CLI, and deploy a VM into it.

Devnets (Developer Networks) are isolated lux networks deployed on the cloud. Similar to local networks in terms of configuration and usage but installed on remote nodes.

Think of DevNets as being an intermediate step in the developer testing process after local network and before Testnet network.

warning

ALPHA WARNING: This command is currently in experimental mode. Proceed at your own risk.

Prerequisites‚Äã

Before we begin, you will need to have:

  • Created an AWS account and have an updated AWS credentials file in home directory with [default] profile
  • Install Terraform
  • Created a Subnet configuration <SubnetName> based on SubnetEVM.

Note: the tutorial is based in AWS, but Devnets can also be created and operated in other supported cloud providers, such as GCP.

Choose a Deployment Method‚Äã

You can either use a sequence of prompt-based commands to get CLI to ask you all the required information for the setup, or execute one command than encompasses all steps, with appropriate command line flags for all the needed info.

For the second case we provide two important use cases examples:

  • creating a devnet an deploy a preexistent CLI Subnet (same example as the step by step)
  • creating a devnet, create a Subnet based on a Custom VM, and deploy it (similar to this one)

Deploying the Subnet into the Devnet‚Äã

At this point, you have a working Devnet, but no Subnet association:

lux node status <clusterName> --subnet <subnetName>

So, next step is to deploy <subnetName> into <clusterName>. So as to create the Subnet and blockchain transactions needed for posterior syncing and validation of the Subnet.

While the subnet deploy command can be used as described here, deploying into Devnets is a subset of that operation, and so a quickest command is provided.

It automatically takes into account the API endpoint of the Devnet, and use a common funding option for Devnets, known as EWOQ keys.

Create a Devnet and Deploy a Subnet into It‚Äã

CLI provides the devnet wiz command that takes care of all the mentioned steps.

As other CLI commands, all the options can be specified by command line flag instead of having the user interactively provide the inputs.

As reference we provide example of the flags setting needed for this tutorial:

lux node devnet wiz <clusterName> <subnetName> --authorize-access\
  --aws --num-nodes 5 --region us-east-1 --default-validator-params

Create a Devnet, Create a Custom VM based Subnet, and Deploy It‚Äã

Prerequisites‚Äã

You will need to have:

  • Created a Custom VM, as described here.

Custom VM‚Äã

The tutorial will create exactly the same Custom VM as here.

Based on the Custom VM tutorial, we will be deploying the TokenVM example built with the LuxSDK.

The following settings will be used:

  • <repoUrl> https://github.com/luxdefi/hypersdk/
  • <branch> main
  • <buildScript> examples/tokenvm/scripts/build.sh

Following the Custom VM tutorial, create the needed input files, and set their paths to:

  • <genesisPath> Genesis File
  • <chainConfigPath> Blockchain Config
  • <subnetConfigPath> Subnet Config
  • <avagoConfigPath> Luxd Config

Execute Wiz‚Äã

Now we are ready to execute the Devnet Wizard. It it going to:

  • create the Subnet configuration, generating also the local VM binary
  • create the Devnet
  • deploy the Subnet into the Devnet
  • make the Devnet nodes to start tracking the Subnet
  • make the Devnet nodes to start validating the Subnet

The command assumes <clusterName> and <subnetName> have not been previously created.

The full command line is:

lux node devnet wiz <clusterName> <subnetName> --custom-subnet \
  --subnet-genesis <genesisPath> --custom-vm-repo-url <repoUrl> \
  --custom-vm-branch <branch> --custom-vm-build-script <buildScript> \
  --chain-config <chainConfigPath> --subnet-config <subnetConfigPath> \
  --node-config <avagoConfigPath> --authorize-access --aws --num-nodes 5 \
  --region us-east-1 --default-validator-params
